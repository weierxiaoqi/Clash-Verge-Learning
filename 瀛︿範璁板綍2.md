## 8. 为特定应用创建专属代理组（以 Cursor 为例）

### 8.1 为什么需要专属代理组？

Cursor 依赖 Anthropic API，其服务器主要位于美国，使用以下节点通常更快更稳定：

| 地区 | 原因 |
|------|------|
| **美国西海岸**（LA、硅谷） | 离 Anthropic 服务器最近，延迟最低 |
| **日本**（东京） | 亚太最稳定，延迟可接受 |
| **新加坡** | 带宽充足，备用选择 |

不建议使用香港、欧洲节点：香港节点可能被 Anthropic 限速，欧洲绕路太远。

---

### 8.2 正确的修改方式：用扩展文件，不动主订阅

Clash Verge 的订阅文件每次更新会被覆盖，**绝对不能直接修改订阅文件**。

正确做法是使用 Clash Verge 的**扩展文件系统**：

| 扩展文件类型 | 作用 | 对应操作 |
|------------|------|---------|
| `groups`（代理组扩展） | 新增/修改代理组 | 在此添加 Cursor 专用组 |
| `rules`（规则扩展） | 新增/修改规则 | 在此添加 Cursor 域名规则 |
| `script`（脚本扩展） | JS 脚本动态修改配置 | 复杂场景使用 |
| `merge`（合并扩展） | YAML 合并覆盖 | 修改全局配置项 |

这些扩展文件挂载在订阅上，**订阅更新不会影响它们**。

---

### 8.3 代理组扩展文件配置（groups 文件）

```yaml
prepend:
  # 原有的 Claude 组保持不变
  - { name: Claude, type: select, proxies: [美国USLA-A, 美国LA-优化-GPT, ...] }

  # 新增：Cursor 专用组
  - name: 🚀 Cursor专用
    type: url-test                          # 自动测速选最快节点
    proxies:
      # 美国节点
      - 美国USLA-A
      - 美国LA-优化-GPT
      - 美国LA-优化2-GPT
      - 美国LA-优化3-GPT
      # 日本节点
      - 日本-优化
      - 日本-优化2
      - 日本-优化3
      - 日本JP-HY2
      - 日本JP-A
      # 新加坡节点
      - 新加坡SG-HY2
      - 新加坡-优化-Gemini-GPT
      - 新加坡-优化2-Gemini-GPT
      - 新加坡-优化3-Gemini
    url: 'https://cp.cloudflare.com/generate_204'  # 测速地址
    interval: 300     # 每 300 秒重新测速
    tolerance: 100    # 延迟差 100ms 内不切换，避免频繁跳节点
    lazy: false       # 启动时立即测试，不等懒加载
```

---

### 8.4 规则扩展文件配置（rules 文件）

```yaml
prepend:
  # Cursor 专用规则（放在最前面优先匹配）
  - DOMAIN-SUFFIX,cursor.sh,🚀 Cursor专用
  - DOMAIN-SUFFIX,cursor.com,🚀 Cursor专用
  - DOMAIN-SUFFIX,cursorapi.com,🚀 Cursor专用

  # 原有 Claude/Anthropic 规则保持不变
  - DOMAIN-SUFFIX,claude.ai,Claude
  - DOMAIN-SUFFIX,anthropic.com,Claude
  # ...
```

**注意：** `anthropic.com` 保留指向 `Claude` 组，因为 Cursor 访问 Anthropic API 的流量走的是 `cursor.sh` 域名，两者互不干扰。

---

### 8.5 url-test 类型详解

`url-test` 是 Clash 的自动测速代理组，行为如下：

```
启动/每隔 interval 秒
       ↓
对组内所有节点发起 HTTP 请求（测试 url 字段指定的地址）
       ↓
能通  → 记录延迟（ms）
超时  → 视为不可用，不会被选中
       ↓
自动选择延迟最低的可用节点
```

**tolerance 参数的作用：**

> 假设当前用的节点延迟 50ms，此时测速发现有个节点延迟 80ms。  
> 差值 = 80 - 50 = 30ms < tolerance(100ms)，**不切换**。  
> 如果差值 > 100ms，才会切换到更快的节点。

这样设计是为了避免网络抖动时节点频繁来回切换，影响连接稳定性。

**与其他代理组类型的对比：**

| 类型 | 行为 | 适用场景 |
|------|------|---------|
| `url-test` | 自动测速，始终用最快节点 | 追求速度，全自动，适合 Cursor |
| `select` | 手动在 UI 选择节点 | 需要固定某个节点时 |
| `fallback` | 按顺序，当前挂了才切下一个 | 追求稳定，允许一定延迟 |
| `load-balance` | 多节点并行，分摊流量 | 大流量下载场景 |

**Cursor 为什么选 url-test 而不是 select？**

Cursor 的响应速度直接影响编码体验，使用 `url-test` 可以在美/日/新 13 个节点中自动挑出当前最快的，不需要手动维护，适合这种对延迟敏感的场景。

---

### 8.6 修改完成后的生效方式

修改扩展文件后，需要在 Clash Verge 中**重新激活订阅**（点击订阅卡片使其生效）。

生效后验证：
1. 在**代理（Proxies）**页面看到 `🚀 Cursor专用` 组出现
2. 组内节点旁边显示延迟数值，超时节点自动排除
3. 在**连接**日志中，访问 `cursor.sh` 时显示走 `🚀 Cursor专用`
