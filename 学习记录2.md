# Clash Verge 核心概念：代理模式 & TUN 模式深度解析

> 面向新手的完整学习文档，重点讲清楚"流量到底怎么走"这件事。

---

## 0. 最重要的前提：一个经常被误解的问题

很多新手开启了 Clash Verge，看到界面在运行，就认为"所有流量都走代理了"。

**这是错误的。**

正确的理解是：

```
Clash Verge 的代理功能分为两个完全独立的问题：

问题一：流量能否进入 Clash？         ← 这由"流量接管方式"决定
问题二：进入 Clash 的流量该如何走？  ← 这才是"三种代理模式"决定的
```

很多人配置出问题，都是因为只想着问题二，却忽略了问题一。

---

## 1. 问题一：流量如何进入 Clash？

### 1.1 两种流量接管方式

Clash Verge 有两种方式接管你电脑上的网络流量：

#### 方式 A：系统代理（System Proxy）

通过修改 Windows 系统的代理设置，告诉操作系统"我有一个本地代理，地址是 `127.0.0.1:7897`"。

```
Windows 注册表中写入：
  ProxyEnable  = 1
  ProxyServer  = 127.0.0.1:7897

         ↓ 只有"主动读取系统代理"的程序才会看到这个设置

支持的程序：浏览器(Chrome/Edge/Firefox) → 流量进入 Clash ✓
不支持的程序：大多数 CLI 工具、游戏客户端  → 流量直接出网 ✗
```

**系统代理的本质：自愿原则。** 程序自己决定要不要遵守系统代理设置。

#### 方式 B：TUN 模式（虚拟网卡接管）

在系统网络层创建一张虚拟网卡（名为 `Meta` 或 `Mihomo`），通过修改系统路由表，强制将所有网络流量导入这张虚拟网卡，再交给 Clash 处理。

```
程序发出的任何流量
       ↓
操作系统内核路由（所有流量都被路由到 TUN 虚拟网卡）
       ↓
Clash 内核接管处理（规则匹配、分流）
       ↓
代理节点 / 直连
```

**TUN 模式的本质：强制原则。** 不管程序愿不愿意，流量都被操作系统层面截获，程序自身感知不到代理的存在。

---

### 1.2 哪些程序走系统代理，哪些不走？

这是新手最容易踩坑的地方，下面列出常见情况：

#### 默认遵守系统代理的程序

| 程序类型 | 说明 |
|----------|------|
| Chrome / Edge / Firefox | 默认读取系统代理 |
| Electron 应用（如 VS Code、Discord） | 大多数遵守 |
| Python `requests` 库 | 遵守 `http_proxy` 环境变量（等效于系统代理） |
| PowerShell `Invoke-WebRequest` | 遵守系统代理 |
| Windows `curl.exe`（较新版） | 部分遵守 |

#### 默认**不**遵守系统代理的程序

| 程序类型 | 原因 | 解决方式 |
|----------|------|---------|
| `cmd.exe` 内的命令 | 不读系统代理注册表 | 手动设置环境变量，或开 TUN |
| PowerShell（部分 CLI 工具） | 工具自己直连 | 手动 `$env:http_proxy` 或开 TUN |
| Git | 不默认走代理 | `git config --global http.proxy` 或开 TUN |
| npm / yarn / pnpm | 不默认走代理 | `npm config set proxy` 或开 TUN |
| Python `urllib` | 部分不走 | 设置环境变量或开 TUN |
| 游戏客户端（Steam、Epic等） | 自己管理网络 | 通常只能用 TUN |
| Java 程序 | JVM 有自己的代理机制 | 需要 JVM 参数或开 TUN |
| Docker | 有自己的网络栈 | 需要 Docker 代理设置或开 TUN |
| UDP 流量（游戏、视频通话） | 系统代理只支持 TCP | 必须用 TUN |

---

### 1.3 如何让 CLI 工具走代理（不开 TUN 的情况下）

如果你不开 TUN，但需要让某个命令行工具走 Clash，可以手动设置代理环境变量：

**临时设置（当前终端会话有效）：**

```bash
# PowerShell
$env:http_proxy  = "http://127.0.0.1:7897"
$env:https_proxy = "http://127.0.0.1:7897"
$env:all_proxy   = "socks5://127.0.0.1:7898"

# CMD
set http_proxy=http://127.0.0.1:7897
set https_proxy=http://127.0.0.1:7897
```

**针对特定工具设置：**

```bash
# Git
git config --global http.proxy  http://127.0.0.1:7897
git config --global https.proxy http://127.0.0.1:7897

# npm
npm config set proxy      http://127.0.0.1:7897
npm config set https-proxy http://127.0.0.1:7897

# pip（在命令里直接指定）
pip install xxx --proxy http://127.0.0.1:7897

# curl（直接指定）
curl --proxy http://127.0.0.1:7897 https://example.com
```

**注意：** 这些方法都只对 TCP 流量有效，且需要针对每个工具单独配置。如果你有大量 CLI 工具需要走代理，开 TUN 模式是更简洁的方案。

---

### 1.4 当前配置对比图

```
【系统代理模式（你当前状态）】

Chrome       → 读取系统代理 → 进入 Clash → 规则分流 → ✓
Edge         → 读取系统代理 → 进入 Clash → 规则分流 → ✓
PowerShell   → 忽略系统代理 → 直接出网            → ✗（除非手动设环境变量）
Git          → 忽略系统代理 → 直接出网            → ✗（除非 git config 设置）
游戏         → 忽略系统代理 → 直接出网            → ✗

【TUN 模式】

Chrome       → 操作系统路由截获 → 进入 Clash → 规则分流 → ✓
PowerShell   → 操作系统路由截获 → 进入 Clash → 规则分流 → ✓
Git          → 操作系统路由截获 → 进入 Clash → 规则分流 → ✓
游戏（TCP）  → 操作系统路由截获 → 进入 Clash → 规则分流 → ✓
游戏（UDP）  → 操作系统路由截获 → 进入 Clash → 规则分流 → ✓
```

## 2. 问题二：进入 Clash 的流量该如何走？（三种代理模式）

弄清楚了"流量如何进入 Clash"之后，我们再来看三种模式。

**重申：这三种模式只控制"已经进入 Clash 的流量"该如何被处理，与流量能否进入 Clash 无关。**

---

### 2.1 规则模式（Rule）— 日常最常用

按配置文件中的规则列表，逐条匹配流量，决定走哪个策略组。

```
流量进入 Clash
       ↓
从第一条规则开始，逐条匹配：

  DOMAIN-SUFFIX,google.com → 命中 → 走「代理」策略组
  DOMAIN-SUFFIX,baidu.com  → 命中 → 走「直连」策略组
  GEOIP,CN                 → 命中 → 直连
  MATCH                    → 兜底 → 走「代理」策略组（兜底规则，所有未命中的都走这里）
```

**规则的优先级：从上到下，先命中先执行。**

常见规则类型：

| 规则类型 | 示例 | 说明 |
|----------|------|------|
| `DOMAIN` | `DOMAIN,api.openai.com,代理` | 精确匹配完整域名 |
| `DOMAIN-SUFFIX` | `DOMAIN-SUFFIX,openai.com,代理` | 匹配该域名及所有子域名 |
| `DOMAIN-KEYWORD` | `DOMAIN-KEYWORD,openai,代理` | 域名中包含关键词 |
| `IP-CIDR` | `IP-CIDR,192.168.0.0/16,直连` | 匹配 IP 段 |
| `GEOIP` | `GEOIP,CN,直连` | 按 GeoIP 库匹配国家 |
| `PROCESS-NAME` | `PROCESS-NAME,steam.exe,代理` | 按进程名匹配（TUN 模式下有效） |
| `MATCH` | `MATCH,代理` | 兜底，匹配所有未命中的流量 |

**适合场景：** 日常使用，国内直连、国外走代理，精细化控制。  
**缺点：** 规则没写全的服务可能走错出口，需要持续维护规则。

---

### 2.2 全局模式（Global）

所有进入 Clash 的流量，全部走同一个全局代理节点，不做任何分流判断。

```
流量进入 Clash
       ↓
忽略所有规则
       ↓
全部走【全局代理】选中的节点
```

**适合场景：**
- 临时测试某个节点是否可用
- 某个服务在规则模式下分流有问题，临时切全局排查
- 需要所有流量必须经过代理的场景

**缺点：**
- 国内网站也会绕道代理节点，速度变慢
- 访问国内服务（如支付宝、微信）可能出现异常
- 节点流量消耗更大

---

### 2.3 直连模式（Direct）

所有进入 Clash 的流量，全部直接出网，不走任何代理节点。

```
流量进入 Clash
       ↓
忽略所有规则和节点
       ↓
全部直接连接目标地址
```

**注意：** 直连模式和"关闭 Clash"不完全一样。流量依然经过 Clash 的端口转发，只是不走代理节点而已。

**适合场景：**
- 临时需要全部直连，但不想关闭 Clash Verge 界面
- 测试"不走代理时网络是否正常"

---

### 2.4 三种模式的关系总结

```
                        规则模式（Rule）
                       ↗               ↘
            流量进入 Clash              按规则走不同出口
                       ↘               ↗
                        全局模式（Global）→ 全部走一个节点
                       ↗
                       直连模式（Direct）→ 全部直连出网
```

用一句话总结：
- **规则模式** = 精细化分流，适合日常
- **全局模式** = 全部代理，适合排查问题
- **直连模式** = 全部直连，相当于临时关代理

## 3. TUN 模式详解

### 3.1 TUN 是什么

TUN（Tunnel）是一种虚拟网络设备技术。Clash 利用它在 Windows 系统中创建一张虚拟网卡，并通过操作系统的路由表，将所有流量重定向到这张虚拟网卡，再交由 Clash 内核处理。

你的电脑上开启 TUN 后，可以在 `ipconfig` 中看到一张类似这样的虚拟网卡：

```
未知适配器 Meta:
   IPv4 地址 . . . . . . . . . . . . : 198.18.0.1
   子网掩码  . . . . . . . . . . . . : 255.255.255.252
   默认网关. . . . . . . . . . . . . : 198.18.0.2
```

这张网卡不是真实的物理网卡，而是 Clash 内核创建的虚拟出口。

---

### 3.2 TUN 模式的工作原理

```
①  程序（任何程序）发出网络请求

②  Windows 路由表：
    目标 IP 0.0.0.0/0 → 发往 Meta 虚拟网卡（198.18.0.2）

③  Clash 内核接收流量，进行：
    - DNS 处理（防止 DNS 泄漏）
    - 规则匹配（按你设置的规则/全局/直连模式处理）
    - 连接节点或直连

④  流量经由真实网卡（如 WLAN）出网
```

整个过程对应用程序完全透明，程序感知不到代理的存在。

---

### 3.3 TUN 模式的两种协议栈（Stack）

在 Clash Verge 的 TUN 设置中，有一个 `stack` 选项，决定虚拟网卡使用哪种 TCP/IP 协议栈：

| 选项 | 名称 | 原理 | 适用场景 |
|------|------|------|---------|
| `system` | 系统栈 | 借用 Windows 内核的 TCP/IP 栈 | 兼容性最好，首选 |
| `gvisor` | 用户态栈 | 在用户空间实现完整 TCP/IP | 与系统隔离，兼容性略差 |
| `mixed` | 混合栈 | TCP 用 system，UDP 用 gvisor | 平衡方案 |

**推荐：** 一般情况下使用 `system`，如果出现兼容问题再尝试 `mixed` 或 `gvisor`。

---

### 3.4 TUN 模式与 DNS 的关系（重要！）

开启 TUN 后，必须正确配置 DNS，否则会出现 DNS 泄漏问题（即：请求走了代理，但 DNS 解析却直接发给国内 DNS，泄露了你访问的域名）。

Clash 提供两种 DNS 增强模式：

#### fake-ip 模式（推荐）

```
程序请求解析 google.com
       ↓
Clash DNS 返回一个假 IP（如 198.18.1.100）
       ↓
程序拿着假 IP 发起连接
       ↓
Clash 截获连接，根据内部映射知道这是 google.com
       ↓
按规则处理，真正的 DNS 解析在节点出口侧完成
```

**优点：** 快速，DNS 不会泄漏，分流精准。  
**缺点：** 假 IP 不真实，某些需要真实 IP 的程序（如 nmap）可能有问题。

#### redir-host 模式

```
程序请求解析 google.com
       ↓
Clash DNS 真实解析域名，返回真实 IP
       ↓
程序拿真实 IP 发起连接
       ↓
Clash 截获连接，尝试还原域名，按规则处理
```

**优点：** 兼容性更好，返回真实 IP。  
**缺点：** 速度略慢，域名还原有时会失败，影响分流准确性。

---

### 3.5 TUN 模式开启步骤

1. 在 Clash Verge 界面，点击左侧「设置」
2. 找到「TUN 模式」开关，打开
3. （可选）点击「TUN 设置」，调整 `stack` 类型
4. 确认界面右上角出现「TUN」标识
5. 检查 `ipconfig` 输出是否出现 `Meta` 虚拟网卡

**注意：** 开启 TUN 模式需要管理员权限，Clash Verge 会自动请求提权，或通过 `clash-verge-service.exe` 系统服务完成。

---

### 3.6 TUN 模式常见问题

#### 开启 TUN 后系统网络断了

可能原因：
- 路由表被异常修改，重启 Clash Verge 或重启电脑
- `stack` 与当前系统不兼容，尝试切换到 `system`

#### TUN 开启了，但某个程序还是不走代理

检查：
- 是否有 IP 直连（程序绕过 DNS，直接使用 IP 地址，此时 fake-ip 模式下无法匹配域名规则，需要用 `IP-CIDR` 规则）
- 是否是 IPv6 流量（部分 TUN 配置没有接管 IPv6）
- 查看 Clash 连接日志，确认该程序的流量是否出现

#### TUN 模式下 DNS 泄漏

检查 `dns_config.yaml` 是否正确配置，确保 `enhanced-mode: fake-ip`，且 `nameserver` 中包含可靠的上游 DNS。

## 4. 系统代理 vs TUN 模式，应该选哪个？

| 对比项 | 系统代理 | TUN 模式 |
|--------|---------|---------|
| 接管范围 | 只有主动遵守系统代理的程序 | 系统所有流量 |
| CLI 工具 | 需手动配置环境变量 | 自动接管 |
| 游戏 / UDP | 不支持 | 支持 |
| DNS 控制 | 不接管 | 完全接管 |
| 配置复杂度 | 简单 | 稍复杂 |
| 需要管理员权限 | 否 | 是 |
| 系统兼容性 | 极好 | 好 |

**推荐策略：**

- **日常上网 + 只用浏览器** → 系统代理就够了
- **需要 CLI 工具走代理** → 开启 TUN，或手动配置环境变量
- **打游戏 / UDP 代理** → 必须开 TUN
- **想要最简单、接管最彻底** → 开 TUN

## 5. 实战场景速查

### 场景一：浏览器能访问 Google，但终端里 curl 失败

**原因：** 浏览器走了系统代理，终端没有。  
**解决：**

```powershell
# PowerShell 临时设置
$env:http_proxy  = "http://127.0.0.1:7897"
$env:https_proxy = "http://127.0.0.1:7897"
```

或者开启 TUN 模式。

---

### 场景二：开了代理，但访问 GitHub 还是慢

**原因：** 规则模式下，GitHub 可能命中了直连规则，或走的节点不够快。  
**排查步骤：**
1. 打开 Clash Verge「连接」面板
2. 在浏览器访问 GitHub
3. 查看连接列表中 `github.com` 命中了哪个策略组
4. 如果命中了「直连」，需要在规则增强文件中添加 GitHub 的代理规则

---

### 场景三：规则模式下某服务正常，切全局后反而出问题

**原因：** 全局模式把所有流量都发往代理节点，国内服务走海外节点可能出错。  
**解决：** 切回规则模式，日常不建议长时间用全局模式。

---

### 场景四：游戏延迟没有变化，代理没生效

**原因：** 游戏流量通常是 UDP，系统代理不支持 UDP。  
**解决：** 必须开启 TUN 模式才能接管 UDP 流量。

---

### 场景五：CLI 登录成功，浏览器也正常，但换了台设备/节点后账号异常

**原因：** 同一账号从不同国家、不同 IP 访问，触发了服务方的风控。  
**解决：** 给该服务（如 Claude、ChatGPT）建立单独策略组，固定到一个稳定节点，不要频繁切换。

## 6. 配置状态速查表

| 查看内容 | 方法 |
|----------|------|
| 当前代理模式（规则/全局/直连） | Clash Verge 主界面右上角 |
| TUN 是否开启 | 设置页面 / `ipconfig` 查看是否有 Meta 网卡 |
| 系统代理是否开启 | Clash Verge 主界面「系统代理」开关 |
| 某个请求走了哪个节点 | Clash Verge「连接」面板实时日志 |

| 端口是否正常监听 | PowerShell：`netstat -ano \| findstr "7897"` |

## 7. 总结：从"开了代理"到"代理真正生效"的完整路径

```
Step 1：确认 Clash Verge 正在运行，内核正常启动

Step 2：选择流量接管方式
  ├── 只需要浏览器走代理 → 开启「系统代理」即可
  └── 需要所有程序/CLI/游戏走代理 → 开启「TUN 模式」

Step 3：设置代理模式（控制进入 Clash 后的流量走向）
  ├── 日常使用 → 规则模式（Rule）
  ├── 排查问题 → 全局模式（Global）
  └── 临时关代理 → 直连模式（Direct）

Step 4：验证是否生效
  ├── 打开 Clash 连接日志，确认目标域名进入了 Clash
  ├── 确认命中了预期的策略组
  └── 用 ifconfig.me 确认出口 IP 和地区正确
```

记住核心原则：**"流量能否进入 Clash"和"进入后如何走"是两件独立的事情，必须分开理解。**
